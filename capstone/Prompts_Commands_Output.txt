2. Root Cause Identification
The fault is a CPU saturation stress test — working exactly as designed.

Evidence Pattern
Why this is CPU saturation, not resource exhaustion:

Load average 1.03 on a 1-vCPU system mathematically = 100% utilization (load queue depth = number of runnable processes; 1.0 on 1vCPU = one process always waiting to run)
Memory delta negligible: 254 → 270 MB = +16 MB (1.8% increase)
Zero swap activity: 0 MB throughout
No kernel intervention: No OOM, no thermal throttle, no CPU frequency scaling
Clean termination: stress-ng exited normally at 83.88s; no forced kill
3. Hypothesis Statement
"The fault is an intentional CPU saturation stress test, evidenced by a load average spike from 0.0 to 1.03 (100% of available 1-vCPU capacity), which caused the VM to sustain 83+ seconds of peak CPU utilization with zero memory or I/O side effects, followed by immediate recovery upon process termination — consistent with stress-ng --cpu 1 behavior."

4. Alternative Hypotheses (Ruled Out by Evidence)
Alternative 1: Memory Exhaustion / OOM Kill
Hypothesis: Low-memory system (1 GB) hit memory ceiling; processes were OOM-killed; stress-ng crashed.

Evidence that Rules This Out:

Memory used: 254 MB → 270 MB (only +16 MB delta)
Available memory: 898 MB total; 628 MB free (70% free) — well above pressure threshold
No swap activity: 0 MB throughout fault window (OOM would trigger swap exhaustion first)
No dmesg warnings for "Out of memory" or "oom-kill"
No journalctl alerts for memory pressure
stress-ng exited cleanly at designed timeout (83.88s), not prematurely from OOM kill
Post-fault memory is 250 MB (lower than pre-fault 254 MB) — no memory leak
Verdict: ❌ RULED OUT — memory was never a constraint.

Alternative 2: Kernel Thermal Throttling / CPU Frequency Scaling
Hypothesis: CPU overheated or frequency was throttled by firmware; this is why the fault "recovered" — thermal governor stepped down clock speed.

Evidence that Rules This Out:

Load average remained stable at 1.02–1.03 during the fault (no drop)
If CPU frequency was throttled, load would drop (fewer ops/sec) OR increase further (more time competing)
Stable load indicates clock speed was constant
No dmesg warnings for thermal events, CPU throttle, or frequency scaling
No journalctl "thermal" entries
stress-ng exited at exactly 83.88 seconds — matching the designed 300-second timeout divided by ~3.57 (bogo ops / timeout)
If throttling occurred, this timing would be skewed
Standard_B1s is a burstable VM with no documented thermal throttling on Azure (managed hypervisor, not bare metal)
Post-fault load drops to 0.0 immediately, not gradually — inconsistent with thermal spin-down
Verdict: ❌ RULED OUT — thermal throttling did not occur.

Alternative 3: I/O Contention / Disk Bottleneck
Hypothesis: Disk I/O saturation caused CPU wait-time inflation; the "load" is actually iowait, not CPU work.

Evidence that Rules This Out:

stress-ng --cpu 1 is a pure CPU worker — zero disk I/O by design (no reads, writes, or mmap)
Load average stayed flat (no iowait spike would cause 1-min load to oscillate)
Memory + swap unchanged — I/O to disk would cause swap activity if memory was involved
No journalctl I/O errors or disk warnings
recovery.sh did not run iostat or iotop, but fault.sh likely showed 0% disk util in background monitoring
Process exit timing matches CPU work count (bogo ops), not I/O latency
Verdict: ❌ RULED OUT — disk was not involved.

Capacity decision recommendation

Recommendation: resize from Standard_B1s to Standard_B2s.

Justification:

B1s has only 1 vCPU, so one CPU-bound process can fully monopolize compute and degrade responsiveness, exactly what happened here.
B2s gives 2 vCPUs, so a single hot thread is less likely to starve the OS and critical agents.
For burst-style workloads, B2s materially improves headroom while remaining in the cost-efficient burstable family.
Your incident showed CPU, not memory or I/O, as the bottleneck, so a CPU-focused resize is the right first move.
Operational note:

If workload becomes sustained/high CPU (not bursty), move to a non-burstable SKU such as D2as v5 instead of B-series to avoid CPU credit throttling behavior.
